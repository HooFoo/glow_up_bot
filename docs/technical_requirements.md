# Технические требования: Prime/Glow Ассистент

## 1. Системные требования

### Сервер
- **OS:** Linux (Ubuntu 22.04 LTS рекомендуется)
- **PHP:** 8.2+
- **MySQL:** 8.0+
- **Nginx:** 1.24+ или Apache 2.4+
- **PHP Extensions:** `pdo_mysql`, `curl`, `mbstring`, `json`, `fileinfo`
- **FPM:** Использование `fastcgi_finish_request()` для неблокирующей обработки
- **HTTPS:** обязателен (Let's Encrypt / Cloudflare)
- **Хранилище:** минимум 10 GB (основной объём занимают логи и временные медиафайлы)

### PHP-зависимости (Composer)
```json
{
    "require": {
        "php": ">=8.2",
        "vlucas/phpdotenv": "^5.5",
        "guzzlehttp/guzzle": "^7.0"
    }
}
```

## 3. Требования к Telegram Bot API

### Используемые типы входящих Update:
| Тип | Назначение |
|---|---|
| `message.text` | Текстовое сообщение |
| `message.voice` | Голосовое сообщение |
| `message.photo` | Фото (+ caption) |
| `message.successful_payment` | Успешная оплата Stars |
| `callback_query` | Нажатие inline-кнопки |
| `pre_checkout_query` | Подтверждение платежа |

### Команды бота:
| Команда | Описание |
|---|---|
| `/start` | Начало / перезапуск (квиз или главное меню) |
| `/menu` | Показать главное меню |
| `/nutrition` | Перейти в режим питания |
| `/cosmetics` | Перейти в режим косметики |
| `/coach` | Перейти в режим коуча |
| `/subscribe` | Информация о подписке |
| `/profile` | Показать свой профиль |

### Inline Keyboard (кнопки):
- Выбор режима (Питание / Косметика / Коуч)
- Кнопка «Купить подписку»
- Кнопки ответов в квизе (варианты)

## 4. Требования к OpenAI API

### Эндпоинты:
| API | Использование |
|---|---|
| `POST /v1/chat/completions` | Генерация ответов (GPT-4o) |
| `POST /v1/audio/transcriptions` | STT для голосовых (Whisper) |
| `POST /v1/chat/completions` с `image_url` | Vision для фото |

### Лимиты и настройки:
- **Max tokens ответа:** 1024 (диалог) / 2048 (обновление профиля и сжатие)
- **Temperature:** 0.7 (диалог) / 0.3 (профиль и резюме)
- **Timeout запроса:** 60 секунд
- **Retry при ошибке:** 2 попытки с задержкой 2 сек

### Построение контекста (в порядке убывания важности):
1. System prompt (из `.md` файла)
2. Профиль пользователя (JSON → text)
3. Резюме предыдущих разговоров
4. Последние 5 сообщений (user + assistant)

## 5. Бизнес-логика: квиз

### Требования:
- Вопросы хранятся в `quiz/questions.md` в структурированном формате
- Каждый вопрос имеет `id`, `text`, `type` (single/multi), `options`
- Ответы сохраняются в БД (таблица `quiz_answers`)
- По завершении квиза — `PersonaService` определяет типаж
- Типажей: **5 штук** (определяются по scoring ответов)
- Каждый типаж имеет: название, описание, рекомендации, изображение
- После квиза устанавливается `quiz_completed_at` и активируется бесплатный период

### Правила квиза:
- Пользователь не может вернуться к предыдущему вопросу
- При `/start` повторно — если квиз уже пройден, переходит в меню
- Квиз нельзя пройти повторно (профиль обновляется AI, не пересдачей)

## 6. Бизнес-логика: Chat & Memory

### Счётчики сообщений:
- `message_count` — общий счётчик всех сообщений пользователя
- **Каждые 5 сообщений:** `SummaryService::compressLast5()` — создаёт/обновляет резюме через LLM
- **Каждые 10 сообщений:** `ProfileService::updateFromHistory()` — обновляет поле `profile_json` через LLM

### Режимы (modes):
- `active_mode` хранится в БД (nutrition / cosmetics / coach)
- При смене режима контекст сохраняется (резюме и профиль общие)
- Системный промпт меняется на соответствующий файл

### Chat Events (Индикация набора сообщения):
- Перед началом тяжелых операций (STT, Vision, LLM-генерация) — отправка `sendChatAction` с параметром `typing` (или `record_voice`, `upload_photo` в зависимости от контекста).
- Так как LLM может генерировать ответ долго, нужно повторять `sendChatAction` каждые 4-5 секунд в цикле, если процесс затянулся (опционально, зависит от реализации).

### Быстрый HTTP ответ (Non-blocking):
- Бот должен возвращать HTTP 200 OK **сразу** после получения и маршрутизации запроса.
- Механизм реализации:
    1. Игнорировать завершение соединения пользователем: `ignore_user_abort(true);`
    2. Установить заголовок `Content-Length: 0` или отправить пустой ответ.
    3. Вызвать `fastcgi_finish_request();` (если используется PHP-FPM).
    4. Продолжить выполнение логики (Background processing).
- Это гарантирует, что Telegram не будет присылать повторные уведомления (retry) из-за таймаута ожидания ответа.

### Голосовые сообщения:
1. Скачать `voice` файл через Telegram `getFile`
2. Отправить на Whisper API → получить текст
3. Временный файл удалить после транскрипции
4. Дальнейшая обработка — как текстовое сообщение

### Фото:
1. Скачать наибольший по размеру `photo` из массива
2. Получить `caption` из сообщения
3. [ChatAction: "upload_photo" или "typing"]
4. Отправить в GPT-4o Vision: `[image_url, caption]` → текстовое описание/анализ
5. Временный файл удалить
6. Сохранить ответ AI в историю сообщений

## 7. Бизнес-логика: подписка и пейвол

### Проверка доступа (`SubscriptionService::checkAccess`):
```
1. Пользователь прошёл квиз? → НЕТ → направить на квиз
2. Есть активная подписка? (subscription_end > NOW()) → ДА → доступ разрешён
3. Бесплатный период? (quiz_completed_at + FREE_DAYS > NOW()) → ДА → доступ разрешён
4. Иначе → пейвол (показать кнопку покупки)
```

### Покупка подписки (Telegram Stars):
1. Пользователь нажимает «Купить подписку»
2. Бот отправляет `sendInvoice` с ценой в Stars
3. Telegram присылает `pre_checkout_query` → бот подтверждает (`answerPreCheckoutQuery`)
4. После оплаты Telegram присылает `successful_payment`
5. `SubscriptionService::activate()` → запись в `subscriptions` + обновление `users.subscription_end`

## 8. Требования к Admin-панели

### Авторизация:
- HTTP Basic Auth или простая сессионная форма логин/пароль
- Credentials из `.env`
- Все страницы защищены

### Dashboard:
- Карточки: всего пользователей, активных подписок
- Line chart (Chart.js): новые регистрации по дням (последние 10 дней)
- Line chart (Chart.js): покупки подписок по дням (последние 10 дней)

### Список пользователей (`/admin/users`):
- Сортировка: сначала с активной подпиской (по дате регистрации DESC), затем без подписки (по дате регистрации DESC)
- Колонки: ID, Username, Имя, Дата регистрации, Статус подписки, Истекает
- Пагинация: 20 пользователей на страницу

### Карточка пользователя (`/admin/users/{id}`):
- Telegram ID, username, first_name, last_name
- Количество сообщений
- Статус подписки, дата начала, дата окончания
- Профиль пользователя (JSON отображается как читаемый текст)
- Резюме переписки

### Переписка (`/admin/users/{id}/chat`):
- Отображение всех сообщений в стиле чата (user слева, bot справа)
- Без медиафайлов (только текст, для голосовых — пометка `[Голосовое сообщение]`, для фото — `[Фото]`)
- Пагинация: 50 сообщений на страницу
- Последние сообщения сверху / снизу: снизу (как в мессенджерах)
